Fiche de Cours : Anatomie Réelle de la Mémoire d'un Processus C

Cette fiche documente l'organisation complète et exacte de la mémoire virtuelle du processus de l'exécutable exo, obtenue via le pseudo-fichier /proc/[PID]/maps sous Linux.

1. Code Source Associé (main.c)

Voici le code C compilé pour l'expérience. Les commentaires indiquent dans quelle zone mémoire chaque élément va se situer lors de l'exécution :

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>  // Nécessaire pour getpid()

// Variable globale initialisée -> Segment .data (rw-p)
int ma_globale = 42; 

int main() {
    // Variable locale -> Segment [stack] (rw-p)
    int ma_locale = 10; 

    // Allocation dynamique -> Segment [heap] (rw-p)
    int *mon_tas = malloc(sizeof(int)); 
    *mon_tas = 99;

    printf("--- PROGRAMME EN COURS --- \n");
    printf("Mon PID est : %d\n", getpid());
    printf("Adresse locale: %p | Adresse tas: %p\n", (void*)&ma_locale, (void*)mon_tas);
    
    // Blocage du processus pour permettre la lecture de /proc/maps
    getchar(); 

    free(mon_tas);
    return 0;
}

3. Analyse des Segments de la Table

Segment / Fichier Permissions Explication et Rôle Système
exo (.text) r-xp Segment de Code. Contient les instructions CPU compilées de ton programme (ex: la logique compilée du main()). Le droit x permet l'exécution, mais l'absence de w empêche le programme de s'auto-modifier.
exo (.data) rw-p Données Initialisées. C'est ici que crèche ta variable globale ma_globale = 42. Elle doit être accessible en écriture (w) car sa valeur peut être modifiée par le code à tout moment.
[heap] rw-p Le Tas. Cet espace est créé et agrandi dynamiquement lors de l'appel à malloc(). C'est là que réside la valeur 99 pointée par mon_tas. Les adresses croissent vers le haut.
libc.so.6 Multiples Bibliothèque Standard du C. Chargée en mémoire partagée. C'est elle qui fournit le code de printf(), malloc() et getchar() utilisé dans ton code. Elle est découpée elle aussi en sections de code (r-xp) et de données (rw-p).
[vvar] / [vdso] r--p / r-xp Mécanismes d'optimisation du Noyau (Kernel). Permettent au processus d'exécuter certains appels système très courants (comme récupérer l'heure système) directement en espace utilisateur, sans subir le coût d'un vrai changement de contexte vers le Noyau.
ld-linux-*** Multiples Le Linker Dynamique / Chargeur. C'est le programme système qui a démarré juste avant ton code pour trouver la libc.so sur le disque, la charger en RAM, et lier les adresses de tes printf() vers celle-ci avant de donner la main à ton main().
Lignes vides rw-p Zones Anonymes. Plages de mémoire RAM allouées par le système qui ne correspondent à aucun fichier sur le disque. Utilisées pour des besoins de stockage temporaire par les bibliothèques ou le système d'exploitation.
[stack] rw-p La Pile d'exécution. Située tout en haut de la mémoire virtuelle (adresses en 7ffda...). C'est là que vit ta variable locale ma_locale = 10. Elle grandit vers le bas (adresses décroissantes) lors des appels de fonctions et se dépile automatiquement lors des return.

4. Concepts Clés pour l'Examen : Réalité Physique vs Illusion Virtuelle

A. Tout cela est-il en RAM ? Les données sont-elles contiguës ?

  • Oui, c'est en RAM : Toutes les zones listées ci-dessus (ton code, la pile, le tas, les bibliothèques) sont obligatoirement chargées dans les barrettes de RAM physiques pour que le processeur puisse y accéder à haute vitesse.
  • Ce n'est PAS contigu physiquement : Dans tes barrettes de RAM, les données sont complètement éparpillées et fragmentées en petits blocs de 4 Ko (appelés Pages Physiques) là où le noyau Linux trouve de la place libre. La pile de ton programme peut côtoyer des données de Firefox ou du système.
  • L'illusion de la Continuité : Les adresses que tu vois dans maps sont des adresses virtuelles. Le système d'exploitation et l'unité matérielle MMU (Memory Management Unit) créent une illusion parfaite pour ton programme : ils lui font croire qu'il dispose d'un espace mémoire continu, isolé et parfaitement ordonné. La Table des Pages de l'OS fait la traduction à la volée entre le virtuel (ordonné) et le physique (éparpillé).

B. Quelle est la taille totale de ton code (.text) ?

Pour calculer la taille de ton segment de code machine (la ligne avec les droits r-xp), il faut soustraire l'adresse virtuelle de début à l'adresse virtuelle de fin :

Adresse de fin : 0x557474e8b000
Adresse de début : - 0x557474e8a000
----------------------------------
Taille en Hexadécimal = 0x1000 bytes

En convertissant 0x1000 de l'hexadécimal vers le décimal, on obtient exactement 4096 octets (soit 4 Ko). Ton code compilé occupe précisément la taille d'une page mémoire standard sous Linux !

C. Que signifie le nombre mystère "1046610" ?

Le nombre 1046610 correspond au numéro d'Inoeud (Inode) de ton fichier exécutable exo sur le disque dur.

  • Sous Linux, un numéro d'inoeud est l'identifiant unique d'un fichier dans le système de fichiers (ext4, etc.).
  • La présence de ce numéro dans maps prouve à ton système d'exploitation que cette zone de la mémoire virtuelle est directement "mappée" (liée) à un vrai fichier stocké sur ton disque, et non à de la mémoire anonyme créée à la volée.
  • Tu peux vérifier ce numéro dans ton terminal en tapant la commande : ls -i /home/etudiant/01_progC/exo. Tu verras s'afficher exactement 1046610 !
